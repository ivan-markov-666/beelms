# Coding Standards

Този документ описва стандартите за писане на код, които трябва да се следват в проекта, за да се осигури консистентност, четимост и лесна поддръжка.

## 1. Форматиране и Линтинг

- **Форматиране**: Използваме `Prettier` за автоматично форматиране на кода. Всички файлове трябва да бъдат форматирани преди commit.
- **Линтинг**: Използваме `ESLint` за статичен анализ на кода. CI/CD pipeline-ът ще изпълнява `pnpm lint` и `pnpm format:check` и няма да позволява сливане на код с грешки.
- **Конфигурация**: Основните правила са дефинирани в `.eslintrc.js` и `.prettierrc` в главната директория на проекта.

## 2. TypeScript

- **Strict Mode**: Всички `tsconfig.json` файлове наследяват базовия `tsconfig.base.json`, в който е активиран `"strict": true`. Това включва `strictNullChecks`, `noImplicitAny` и други.
- **Типове**: Предпочита се използването на `interface` за дефиниране на публични API-та и `type` за вътрешни типове, props и state.
- **Споделени Типове**: Всички типове, които се споделят между frontend и backend, трябва да се намират в `packages/shared-types`.

## 3. Naming Conventions (Конвенции за именуване)

- **Файлове**: `kebab-case.ts` (напр. `user.service.ts`). За React компоненти се използва `PascalCase.tsx` (напр. `UserProfile.tsx`).
- **Променливи и Функции**: `camelCase`.
- **Класове и Интерфейси**: `PascalCase`.
- **Компоненти (React)**: `PascalCase`.
- **Константи**: `UPPER_SNAKE_CASE` (напр. `API_ROUTES`).

## 4. API и DTOs

- **DTOs**: Всички Data Transfer Objects за API заявки и отговори трябва да се дефинират в `packages/shared-types` и да използват `class-validator` декоратори за валидация.
- **API Отговори**: Грешките трябва да следват стандартизиран JSON формат, който се обработва от глобалния `ExceptionFilter`.

## 5. Git и Commits

- **Commit Съобщения**: Препоръчително е да се следва [Conventional Commits](https://www.conventionalcommits.org/en/v1.0.0/) спецификацията.
  - `feat:` (нова функционалност)
  - `fix:` (поправка на бъг)
  - `docs:` (промени в документацията)
  - `style:` (форматиране, липсващи запетаи и др.)
  - `refactor:` (промени в кода, които не поправят бъг и не добавят функционалност)
  - `test:` (добавяне или поправка на тестове)
  - `chore:` (промени по build процес, зависимости и др.)
- **Branches**: Именувайте бранчовете описателно, например `feat/user-registration` или `fix/login-bug`.

## 6. Коментари

- Коментирайте кода само когато е необходимо да се обясни **защо** нещо е направено по определен начин, а не **какво** прави. Кодът трябва да бъде самоописателен.
