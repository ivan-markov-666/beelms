## Основната идея на структурата

Представете си проекта като град, където всеки район има своя функция. Ние ще създадем градската планировка на нашата система. Основните "райони" ще бъдат:

1. **Главният контейнер** - това е като площада на града, където всички сервиси се срещат
2. **Микросервисните квартали** - всеки сервис живее в свой район
3. **Инфраструктурният район** - където живеят базата данни и Redis
4. **Споделените ресурси** - общите площи като парковете

## Директорна структура на най-високо ниво

```
project-root/
├── docker/                 # Докер конфигурации - нашата "инженерна" зона
├── src/                    # Сорс кода - градските квартали
├── shared/                 # Споделени ресурси - общите площи
├── tests/                  # Тестове - инспекционното департамент
├── docs/                   # Документация - библиотеката на града
├── scripts/                # Помощни скриптове - градската логистика
└── data/                   # Данни за разработка - градската памет
```

## Подробно разглеждане на всяка секция

### docker/ - Инженерната зона

Тук живеят всички Докер конфигурации. Всеки файл има специфична роля:

```
docker/
├── Dockerfile.base         # Базовия образ - фундамента на всички сгради
├── services/               # Специфични конфигурации за всеки сервис
│   ├── auth/
│   ├── user/
│   ├── course/
│   └── ... (други микросервиси)
├── nginx/                  # Конфигурация на гейтуея
├── database/               # Конфигурации за PostgreSQL
└── redis/                  # Конфигурации за Redis
```

### src/ - Градските квартали (микросервиси)

Всеки микросервис е като самостоятелен квартал с всичко необходимо:

```
src/
├── auth-service/           # Квартал на автентикацията
│   ├── src/
│   │   ├── controllers/    # Точките за вход - магазините
│   │   ├── services/       # Бизнес логиката - фабриките
│   │   ├── models/         # Структурите на данни - складовете
│   │   ├── middleware/     # Охраната
│   │   └── utils/          # Помощни инструменти
│   ├── tests/              # Инспекцията
│   └── config/             # Местното управление
├── user-service/
├── course-service/
└── ... (други сервиси)
```

### shared/ - Общите площи

Тук са ресурсите, които всички сервиси могат да използват:

```
shared/
├── types/                  # TypeScript типове - общия език на града
├── constants/              # Константи - общите правила
├── utils/                  # Общи помощни функции - общите инструменти
├── interfaces/             # Интерфейси за комуникация
└── guards/                 # Общи охранители за сигурност
```

### tests/ - Инспекционния отдел

Организирани са по типове, за да е лесно да проверяваме всичко:

```
tests/
├── unit/                   # Тестове на компоненти
├── integration/            # Тестове на връзките
├── e2e/                    # Тестове на цялата система
└── fixtures/               # Тестови данни - като филиал на града
```

## Важни принципи за структурата

1. **Самостоятелност**: Всеки микросервис трябва да може да съществува сам, да се разработва и тества независимо.

2. **Минимален износ**: Всеки сервис има само необходимото за работата си - минимизираме зависимостите.

3. **Ясна йерархия**: Като в добре организиран град, всичко има своето място.

4. **Възможност за мащабиране**: Когато "населението" на града нарасне (повече потребители), трябва да е лесно да добавяме "квартали".

5. **Изолация**: Проблем в един "квартал" не трябва да спре целия град.
